# 产品文档评审 - 详细检查清单

需要更细的检查项或示例时使用本文档。

## 1. 完整性检查（细项）

### 产品背景说明
- [ ] 业务/产品背景是否交代（为何做、解决什么问题）
- [ ] 目标用户或角色是否明确
- [ ] 与现有产品/模块的关系是否说明（独立功能 / 延展 / 替代）

### 用户痛点描述
- [ ] 当前痛点或现状是否具体（谁、在什么场景、遇到什么问题）
- [ ] 期望状态或成功标准是否可识别

### 功能需求列表
- [ ] 功能是否以列表或结构化方式呈现
- [ ] 每条是否可理解为独立可交付单元（避免模糊的“优化体验”等）

### 优先级说明
- [ ] 是否有 P0/P1/P2 或类似标注
- [ ] 优先级依据是否合理（业务价值、依赖、风险）

### 验收标准（可选）
- [ ] 若有，是否可执行、可验证（可测试的通过条件）
- [ ] 是否覆盖正常与关键异常场景

---

## 2. 功能需求质量（每一条功能）

对每个功能点逐条核对：

### 边界定义
- **包含**：该功能负责的范围、入口、出口
- **不包含**：明确排除的范围或由其他需求负责的部分
- **与相邻功能**：与上下游或并列功能的接口/责任划分

**反例**：只写“支持导出”，未说明导出格式、数量限制、权限、与现有导出的关系。

### 正常流程
- 触发条件（谁、在什么页面/状态下操作）
- 主要步骤（用户操作与系统响应）
- 结束状态与结果（成功时的界面/数据/提示）

### 异常流程
- 失败场景：网络失败、超时、服务不可用
- 权限/状态：无权限、未登录、数据不存在、状态不允许
- 输入异常：必填缺失、格式错误、超出限制
- 并发/冲突：多人编辑、重复提交等（若适用）

### 前置条件
- 依赖的接口、配置、开关
- 依赖的其他功能或数据（如先有某配置才能用）
- 依赖的环境或权限（角色、租户等）

---

## 3. 设计不合理与优化（常见类型）

评审时可对照以下类型指出问题并给建议：

| 类型 | 说明 | 优化方向 |
|------|------|----------|
| 范围模糊 | 功能边界不清、与现有功能重叠或缺口 | 补充边界定义、与现有功能对比说明 |
| 流程缺失 | 只写主流程，无异常或分支 | 补充异常流程与错误提示策略 |
| 依赖未写 | 未列出前置条件或接口依赖 | 列出依赖与上线顺序 |
| 优先级不清 | 无优先级或与依赖关系矛盾 | 按依赖与价值重排优先级 |
| 验收不可测 | 验收标准主观或不可验证 | 改为可测试的通过条件 |
| 与规范冲突 | 与项目命名、API、权限、RTL 等约定不符 | 按项目规范改写或标注例外 |

---

## 4. 功能与排期（对照代码时）

- **新建**：当前代码无对应实现，需从零开发。
- **改造现有**：在现有模块/页面/接口上修改或扩展；注明文件或模块路径。
- **无直接对应**：文档有描述但代码中无明确对应（可能属后端、配置或未实现）；标注“待确认”或建议落地方向。

排期时优先考虑：依赖关系（前置功能先做）、优先级、单点工作量与风险（可标“待评估”）。
